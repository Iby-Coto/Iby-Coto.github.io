Birthday Paradox project

This project is all about illustrating the famous “birthday paradox. How likely it is that a group of people will share the same birthday. When you run the simulator, it tries out different group sizes (from 5 up to 100) and, for each size, runs 20 separate trials. Each trial creates random birthdays for everyone in that group and checks if any two people have the same date. The simulator gathers all the results, calculates how often a shared birthday actually appears, and prints a summary. You will see how, even with surprisingly small groups, there is a pretty good chance that at least two people share a birthday.

I collaborated on this project with a classmate, I mainly worked on implementing the Trial class and creating test cases. We adopted a lightweight approach to test-driven development (TDD) where I would write a few preliminary tests to confirm that everything from the constructor’s initialization to the detection of duplicate birthdays behaved correctly, including edge scenarios like handling zero people. This process helped catch small errors early and ensured the Trial class was solid before integrating it. We also did occasional pair programming, where one person wrote a quick test while the other tackled the implementation, making troubleshooting faster and more efficient. Working closely together taught me the importance of clear communication and teamwork. This was my first real introduction to using test cases, and it boosted both my confidence and my ability to ensure code reliability from the outset. It was also my first time forking, creating branches, creating pull request and working with others on GitHub.

Overall, this project was both enjoyable and educational, helping me sharpen my Java coding skills and solidifying the value of collaboration in producing robust, well-tested software.

Java Card Game Simulator

The system is designed with flexibility in mind, supporting multiple games through abstraction, delegation, and polymorphism all of which follow OOP and SOLID principles. What the Code Does

At its core, the simulator allows users to play card games in the terminal. It currently supports Go Fish (Human vs AI) and includes placeholders for future expansions like Blackjack and Rummy. The application handles deck creation, shuffling, drawing, tracking drawn cards, managing players' hands, enforcing game rules, and printing game outcomes — all while remaining fully decoupled and extensible.

The game begins via the Simulator> class, which accepts a game name ("gofish") as a command-line argument. It then initializes:

  • A shuffled deck via StandardDeckFactory
  • One human and one AI player (each as a Participants object)
  • A DeckTracker to monitor drawn cards
  • A game-specific behavior strategy (GoFishGameBehavior)
  • And runs the game loop using the CardGame engine

Architectural Structure

The system follows several well-known design patterns:

  • Strategy Pattern: Used to encapsulate different game rules (GameBehaviors) and player strategies (PlayerBehaviors) like AI vs Human
  • Factory Pattern: Used to encapsulate deck creation logic (DeckFactory, StandardDeckFactory)
  • Interface Segregation: All game elements (cards, players, games, decks) are abstracted via interfaces to maximize modularity
  • Single Responsibility: Each class has a clearly defined role, e.g. Participants only manages player hands
  • Open/Closed Principle: New games can be added without modifying the existing engine — only by injecting new behavior implementations

Project

  • DeckOfCards class supports generic playing card management: draw, shuffle, check size
  • Participants hold their own hand and delegate behavior to injected PlayerBehaviors
  • DeckTracker tracks which cards have been drawn
  • GoFishGameBehavior manages game setup and determines a winner
  • GoFishPlayerBehavior implements AI decision-making
  • GoFishHumanPlayerBehavior prompts the user for input to simulate a human turn
  • Mockito is used in all unit test files to isolate behaviors and validate logic flow independently
